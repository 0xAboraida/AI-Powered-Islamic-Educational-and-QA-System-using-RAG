# هيكلية التبديل التلقائي الاحتياطي (LLM Fallback Architecture)

هذا المستند يشرح الهيكلية المتقدمة التي تم تطبيقها في محرك RAG الخاص بمشروع Zad-AI لضمان استقرار النظام بنسبة 100% ضد أعطال واجهات برمجة التطبيقات (APIs) وتحديداً أخطاء خوادم Google Gemini.

## 1. المشكلة (The Problem)
نموذج `Gemini-2.5-Flash` يقدم أداءً ممتازاً، لكن استخدام النسخة المجانية (Free Tier) الخاصة بـ Google يجعله عُرضة لعدة مشاكل:
- **Rate Limits:** قيود على عدد الطلبات في الدقيقة (RPM) أو في اليوم (RPD).
- **أوقات الذروة:** يتم تقليل أولوية الاستجابة في حال وجود ضغط عالمي، مما يؤدي إلى ظهور رسالة الخطأ `503 UNAVAILABLE` أو رسائل مشابهة تفيد بأن الخادم مشغول.
- في السابق، كان هذا الخطأ يؤدي إلى تعطل الـ Pipeline بالكامل وفشل تقديم إجابة للمستخدم.

## 2. الحل: المعمارية الذكية للتبديل (The Solution: Smart Fallback)
تم تصميم نظام وسيط للتبديل التلقائي (Automatic Fallback System) يعتمد على استراتيجية الـ **Primary-Secondary**.
تم اختيار **Groq API** باستخدام نموذج `llama3-70b-8192` ليكون المولد الاحتياطي. 

**لماذا Groq؟**
- يعمل على معالجات (LPUs) مخصصة تعطيه سرعة خرافية (أكثر من 800 كلمة في الثانية).
- نموذج `Llama 3 70B` ذكي جداً ويدعم اللغة العربية بقوة شديدة ويستطيع تعويض Gemini بشكل ممتاز.

## 3. كيف يعمل النظام برمجياً (How it works under the hood)
هذه الاستراتيجية مطبقة في مكانين رئيسيين:
1. **مرحلة معالجة السؤال (Query Preprocessing):** حيث يتم استخدام `with_structured_output` لاستخراج הـ Metadata وتحليل السؤال.
2. **مرحلة توليد الإجابة (Generation):** حيث يتم استخدام `astream` لتوليد الإجابة النهائية.

مسار التوليد يسير كالتالي:

1. **الطلب الأساسي:** يحاول الـ Pipeline تمرير الـ Prompt إلى `Gemini`.
2. **اكتشاف الخطأ (Error Catching):** العملية محاطة بكتلة `try/except`. إذا قام Gemini بإرجاع أي خطأ أو انقطع الاتصال أو أرجع خطأ 503/401...
3. **التنبيه الفوري:** يقوم النظام بالتقاط الخطأ فوراً (وفي مرحلة توليد الإجابة يرسل تنبيهاً مباشراً للمستخدم).
4. **التبديل (Failover):** يتم استدعاء نموذج Groq كاحتياطي.
5. **استكمال الإجابة:** يستلم Groq نفس الـ Prompt ويقوم بالمعالجة (أو توليد الإجابة) بشكل فوري دون أن يتأثر الـ Uptime الخاص بالتطبيق.

## 4. إعدادات النظام (Configuration)
لكي يعمل نظام الـ Fallback، يجب التأكد من وجود هذه المتغيرات في ملف `.env`:

```env
# ========================================================
# ==================== FALLBACK LLM SETTINGS =============
# ========================================================
GROQ_API_KEY=gsk_.............................
FALLBACK_MODEL_NAME=llama3-70b-8192
```

## 5. ميزات هذه الهيكلية
- **الاستقرار (High Availability):** نسبة توافر النظام أصبحت شبه 100% لأن احتمال تعطل سيرفرات Google و Groq في نفس الثانية هو شبه معدوم.
- **تجربة المستخدم (UX):** المستخدم لن يرى أي رسائل برمجة (Errors) أو أعطال، بل سيرى رسالة ودية تشرح له أن النظام ذكي كفاية ليقوم بالتحويل لخادم آخر لخدمته.
- **قابلية التوسع (Scalability):** بفضل نمط المصنع (`factory.py`) والـ `BaseLLMModel`، يمكننا مستقبلاً إضافة OpenAI أو Anthropic كـ Fallback ثالث بسلاسة تامة.
